The Problem

Businesses are drowning in unstructured data

Every day, companies receive thousands of support tickets, reviews and survey responses — yet the meaningful "why" stays buried in plain text.

🏷️

Manual Tagging

Support teams spend hours manually categorising tickets that could be handled in milliseconds.

🐛

Hidden Bugs

Product teams miss emerging issues because they are scattered across thousands of free-text responses.

📉

Late Churn Signals

Customer success teams only discover unhappy customers after they have already left.

Traditional relational databases search by keyword — not by meaning. The result: actionable insight is invisible.

The Solution

Ask your data anything, in plain English

NeuroDash converts every feedback item into a semantic vector embedding and stores it in Aurora. Natural language queries find the most relevant results — even when no keyword matches exist.

Semantic Search
// Query: "checkout confusion"
SELECT content, similarity
FROM feedback_items
ORDER BY embedding <=> $query_vector
LIMIT 10;

Schema Design

Drizzle ORM with custom vector(1536) type mapping directly to pgvector. Type-safe queries pushed to Aurora.

Embedding Pipeline

Each feedback item → Bedrock InvokeModel → 1536-dim float array → stored as pgvector column alongside the original text.

Semantic Query

User query → same embedding pipeline → ORDER BY embedding <=> $vector cosine distance — sub-second results.

Challenges We Ran Into

Real problems, real solutions

Challenge

CloudFormation Engine Version

Aurora PostgreSQL 15.4 wasn't available in eu-central-1 — deployment failed silently.

Fix

Dynamic Version Discovery

Used describe-db-engine-versions to find the latest available version (15.17) and updated the IaC template.

Challenge

VPC Missing Public Routes

New VPC had no route table connecting subnets to the internet gateway — Aurora was unreachable from Vercel.

Fix

IaC Route Table Resources

Added PublicRouteTable, PublicRoute, and subnet associations to CloudFormation template — zero downtime stack update.

Challenge

Native Binaries on Vercel

fastembed + onnxruntime-node had native C++ binaries that silently crashed Vercel's build, returning edge-level 404.

Fix

Switched to Bedrock

Replaced local ONNX model with AWS Bedrock Titan — pure JavaScript SDK, no native binaries, aligns perfectly with the AWS-first stack.

What We Learned

Key takeaways

🐘

pgvector is production-ready

Aurora + pgvector eliminates the need for a dedicated vector database (Pinecone, Weaviate, Qdrant). For most B2B workloads, it delivers equivalent semantic search performance at a fraction of the operational cost.

🤖

Bedrock simplifies AI auth

Using AWS Bedrock instead of OpenAI removes the per-token billing complexity and integrates natively with IAM — no separate API key management, everything stays inside the AWS trust boundary.

🏗️

IaC catches hidden issues

Writing CloudFormation from scratch exposed networking gaps (missing route tables, PubliclyAccessible on wrong resource) that would have been invisible with console-based deployments.

📦

Dependency hygiene matters

Native binary packages (onnxruntime-node) silently break Vercel deployments with a 404 — never leave unused packages in package.json, even after switching providers.
